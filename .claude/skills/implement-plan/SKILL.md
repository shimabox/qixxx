# implement-plan: プラン駆動オーケストレーション

あなた(このセッションのモデル)は **オーケストレーター兼レビュアー** である。実装計画は `docs/plan.md`、レビュー基準はその §6(受け入れ基準)・§7(テスト戦略)・§9(レビュー観点)。

## 絶対ルール

1. **オーケストレーターは `src/`・`tests/`・設定ファイル(vite.config.ts 等)を Edit/Write しない。** 実装・修正はすべて Agent ツールのサブエージェントに行わせる。レビュー指摘の修正も自分で直さず、サブエージェントに差し戻す。
2. オーケストレーターが直接行ってよいのは: ファイルの Read / grep、git 操作(init・ブランチ作成・コミット)、`npm run lint` / `npm run typecheck` / `npm test` / `npm run build` / `npm run e2e` の実行、`docs/` の更新。
3. マイルストーンは docs/plan.md §6 の順に1つずつ。複数マイルストーンを同時に進めない。
4. push / PR 作成・GitHub Pages 設定はユーザーに確認してから行う。
5. サブエージェントの自己申告を信用しない。検証は必ずオーケストレーターが自分のコマンド実行と diff 読みで行う。

## マイルストーン実行手順

### 0. 初回のみ(リポジトリ準備)

- git リポジトリでなければ `git init` し、`docs/`・`.claude/` と .gitignore(node_modules, dist)を初期コミットする
- 対象マイルストーンは、引数があればその番号、なければ git log のコミットメッセージ(`M<N>`)と docs/plan.md §6 から次を判断する

### 1. 準備(オーケストレーター)

- docs/plan.md の該当マイルストーン行(§6)と、関係する仕様節(§3〜§5)・テスト節(§7.1)を読む
- `main` から `feature/m<N>-<短い説明>` ブランチを作成する(既に存在すれば checkout)
- M1 以降は開始前に `npm run lint && npm test` が green であることを確認する

### 2. 実装指示(サブエージェント起動)

プロンプトには必ず以下を含める:

- **役割**: 「docs/plan.md のマイルストーン M<N> を実装する。docs/plan.md 全体と本指示を読んでから着手すること」
- **スコープ**: §6 の該当行の作業内容 + 関係仕様節の番号。**該当マイルストーン以外の変更は禁止**
- **設計上の不変条件**(毎回明記する):
  - `src/core/` は DOM・Canvas・window・AudioContext に依存しない純ロジックにする(§4.4)
  - チューニング定数は `src/config.ts` に集約する
  - ゲームループは固定タイムステップ(§4.4)
  - 原作の名称(QIX/Sparx/Fuse)・見た目をそのまま使わない(§1)
- **完了条件**: `npm run lint` / `npm run typecheck` / `npm test` / `npm run build` が green + §6 の受け入れ基準 + §7.1 の該当テストを実装済み。**コミットはしないこと**(コミットはオーケストレーターが行う)
- **禁止事項**: 新しいランタイム依存の追加(ゲームエンジン・音源ライブラリ含む)、docs/plan.md の書き換え、既存テストの削除・expect の緩和
- **報告形式**: 最終メッセージに「変更ファイル一覧 / 設計上の判断とその理由 / plan.md から逸脱した点(あれば) / テスト結果」を書くこと

### 3. 検証(オーケストレーター)

サブエージェント完了後、自分で以下を実行・確認する:

- `npm run lint && npm run typecheck && npm test && npm run build`(E2E 導入済みの M5 以降は `npm run e2e` も)
- `git diff` を読み、docs/plan.md §6 の該当受け入れ基準と §9 のレビュー観点を1項目ずつ照合する
- 機械的チェック: `grep -rn 'document\.\|window\.\|AudioContext\|getContext' src/core/` がヒットしないこと
- 特に確認する観点:
  - スコープ外のファイルが変更されていないか
  - §7.1 のテストケースが該当分だけ揃っているか(claimArea は文字列フィクスチャ形式か)
  - テストが実装に合わせて弱められていないか(expect の削除・緩和・skip に注意)
  - マジックナンバーが config.ts の外に散っていないか
- 動作確認: `npm run dev` を起動し、受け入れ基準の操作(面取り・ミス・クリア等)を可能な範囲で確認する。ブラウザ操作ツールが使えるならスクリーンショットで確認する

### 4. 差し戻しループ

指摘がある場合は SendMessage で **同じサブエージェントに** 指摘一覧(ファイル:行 + 期待動作)を送り修正させる(コンテキストを保つため新規エージェントにしない)。修正後は手順3を再実行。3回失敗でエスカレーション(モデル選定表参照)。

### 5. マイルストーン完了

- 受け入れ基準・レビュー観点の全通過後、マイルストーン単位でコミットする(メッセージ例: `feat: M1 - ライン引きとエリア確定`)
- ユーザーに報告: 実施内容の要約、レビューで差し戻した指摘と解消結果、次マイルストーンの提案(使用予定モデル込み)
- push / PR / main へのマージはユーザーの指示を待つ

## 進捗管理

- マイルストーンの完了状態は git log(コミットメッセージの `M<N>`)から判断する
- 複数マイルストーンを続けて依頼された場合も、マイルストーン間で必ず手順3〜5(検証・コミット・報告)を挟む
- plan.md との乖離が生じたら、実装を plan.md に合わせるのが原則。plan.md 側を直すべきと判断した場合はユーザーに確認してから docs/plan.md を更新する
