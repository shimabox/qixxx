# QIXXX — QIX クローンゲーム 実装計画書

`docs/first-plan.md` の要件に基づくプランニング結果。本書は実装を担当するモデルへの指示書であり、レビュー時のチェックリストの基準にもなる。

- **実装しないこと**: 本書は計画のみ。コード例は「実装例」として参考提示に留める
- **ゴール**: ブラウザ（スマホ・デスクトップ）で動作する、オリジナルに近いが「似すぎない」QIX クローン

## 1. プロダクト方針 —「いい塩梅」の定義

オリジナルの **ゲームメカニクスは忠実に**、**表現（名称・ビジュアル・数値）はオリジナル独自に** する。

| 項目 | オリジナル QIX | 本作 QIXXX |
|---|---|---|
| タイトル | QIX | **QIXXX**（独自名） |
| 敵（線状生物） | Qix | 独自名に変更（例: **Wisp** / 実装時に決定） |
| 外周の敵 | Sparx | 独自名に変更（例: **Spark**系以外、例: **Ember**） |
| 線の根元から追う敵 | Fuse | 独自名に変更（例: **Igniter**） |
| ビジュアル | 黒背景 + 線描画 | ネオン系の独自配色・グロー表現など、雰囲気は継承しつつ配色/エフェクトは独自 |
| 効果音 | 原作 SE | **Web Audio API による独自生成 SE**（外部音源アセットは使わない） |
| 数値（得点・占有率） | 原作値 | 独自チューニング（後述） |

メカニクス（ライン引き・面取り・低速/高速・占有率クリア・3種の敵）は原作準拠。名称・見た目・数値で差別化する。

## 2. 技術スタック（推奨・変更可）

| 領域 | 選定 | 理由 |
|---|---|---|
| 言語 | **TypeScript** (strict) | コアロジックの型安全性、テスト容易性 |
| 描画 | **Canvas 2D API**（フレームワーク・ゲームエンジンなし） | 線と塗りが主体の本作に十分。依存最小 |
| ビルド | **Vite** | 開発サーバ・ビルド・TS が即動く |
| ユニットテスト | **Vitest** | Vite と親和。コアロジックは DOM 非依存で書くため node 環境で高速に回る |
| E2E | **Playwright**（スモークのみ） | 起動・描画・基本操作の確認 |
| Lint/Format | ESLint + Prettier（または Biome） | 実装モデルの出力品質の底上げ |
| デプロイ | **GitHub Pages**（確定） | 静的ファイルのみ。GitHub Actions で main への push 時に自動デプロイ。`vite.config.ts` の `base` をリポジトリ名に合わせること |
| 効果音 | **Web Audio API**（追加ライブラリなし） | SE を実行時生成。音源アセット不要 |

**依存を増やさないこと。** ゲームエンジン（Phaser 等）は使わない。コアロジックが独立している方がテストとレビューがしやすい。

## 3. ゲーム仕様

### 3.1 フィールドとマーカー（自機）

- フィールドは矩形。内部座標は **論理グリッド**で管理する（推奨: **幅 160 × 高さ 120 セル**、描画時に整数倍スケール）
- マーカーはグリッドの格子点上を移動する
- マーカーは以下の場所のみ移動可能:
  - **境界線上**（フィールド外周 + 確定済みエリアの境界）: 自由に移動
  - **未確定領域内**: ラインボタンを押しながらのみ進入でき、軌跡がラインになる

### 3.2 ライン引きとエリア確定

- ラインボタンは **高速** / **低速** の 2 種。低速は移動速度が半分
- ラインを引き始めた点から、別の境界線上の点に到達するとラインが閉じ、エリアが確定する
- **エリアは「QIX がいない側」にできる**。ラインで未確定領域が 2 分割され、QIX がいない側が塗り潰される
- 低速で囲んだエリアは高得点色（例: 赤系）、高速は通常色（例: 青系）で塗り分ける
- ライン引きの途中で速度を切り替えた場合、**最終的に高速で囲んだ扱い**（原作準拠）
- 引き途中のラインは自分自身と交差できない。交差する移動入力は無視する
- **引き戻し（リトラクト）**: 引いたラインを逆順に戻れる（最後尾のラインセルを削除しながら後退）。ヒューズ（後述）出現後は不可。ライン開始点まで戻り切ると通常の境界線移動に復帰する — **v1 必須**

### 3.3 占有率とステージクリア

- 占有率 = 確定エリア面積 ÷ フィールド内部総面積
- ステージごとの要求占有率に達したらクリア（**デフォルト 65%**。原作は 75% だが、初見ユーザー向けに緩和。設定で変更可能な定数にする）
- クリア時: 超過分をボーナス加点し、次ステージへ

### 3.4 敵

1. **QIX（線状生物）** — 未確定領域内を漂う。複数の線分が残像のように連なった表現（過去 N フレームの線分を描画）。領域内でランダムウォーク+壁反射で移動。**引いている途中のラインまたはマーカーに触れたらミス**。確定済み境界線上のマーカーには無害
2. **スパークス相当（外周の敵）** — 一定時間ごとに外周上部から 2 体出現し、境界線上を移動してマーカーを追う。**マーカーに接触したらミス**。通常は確定済み境界線のみ移動。ステージ内で規定数（例: 3 世代目）以降は引き途中のラインにも進入する（原作準拠だが v1 ではオプション）
3. **ヒューズ相当（導火線）** — ライン引きを **一定時間（例: 1 秒）静止**すると、ラインの根元から出現しライン上をマーカーへ向かって進む。マーカーが動いている間は停止、静止すると再び進む。**追いつかれたらミス**。エリア確定で消滅

### 3.5 ミスとライフ

- ミス条件: QIX が引き途中ラインに接触 / スパークスに接触 / ヒューズに追いつかれる
- ミス時: 引き途中のラインは消滅し、マーカーはライン開始点に戻る。ライフ -1、得点倍率リセット
- 初期ライフ 3。0 でゲームオーバー

### 3.6 スコアリング（独自チューニング値）

- エリア確定: `面積(セル数) × 0.5 点 × 倍率`（高速）/ `× 1.0 点 × 倍率`（低速）— 端数切り捨て
- ステージクリア時: `(達成占有率 − 要求占有率) × 100 点` の超過ボーナス
- **2 匹 QIX の分断倍率（v1 必須）**: 2 匹の QIX がいるステージで、両者を別々の領域に分断するラインを引くとそのステージは即クリア。次ステージから得点倍率が `分断成功回数 + 1` 倍になる（**上限 9 倍**）。1 度でもミスすると倍率は 1 倍に戻る（原作準拠）

### 3.7 ステージ進行（難易度カーブ）

| ステージ | QIX | QIX 速度 | スパークス出現間隔 | 要求占有率 |
|---|---|---|---|---|
| 1 | 1 匹 | 基準 | 30 秒 | 65% |
| 2 | 1 匹 | ×1.15 | 25 秒 | 65% |
| 3+ | **2 匹**（分断クリア可能） | 逓増（上限 ×2） | 逓減（下限 10 秒） | 65〜75% へ逓増 |

2 匹ステージでは、通常クリア（占有率達成）と分断クリアの両方が可能。通常の面取りでは「どちらの QIX からも到達できない領域」だけが塗られる。

数値は定数ファイル（`config.ts`）に集約し、チューニング可能にする。

### 3.8 効果音（v1 必須）

- **Web Audio API で SE を実行時生成**する（オシレーター + エンベロープ）。外部音源ファイルは使わない（原作との差別化 + アセット管理不要）
- 対象イベント: ライン引き中（速度で音程差）/ エリア確定 / ミス / ステージクリア / 分断クリア（特別音）/ ヒューズ出現・接近 / スパークス出現
- モバイルの自動再生制限対策: 初回タップ/キー入力で `AudioContext.resume()` する
- ミュートボタンを HUD に置く（設定は localStorage 保存）

### 3.9 スコープ外（v1 では実装しない）

- エリア頂点からの分岐路テクニック（原作の高等テク）
- 原作のバグ再現（99% クリア、ジグザグでスパークスを撒く等）
- BGM（SE のみ。BGM は v2 検討）
- ハイスコアのサーバ保存（localStorage のみ）

## 4. コア設計

### 4.1 フィールドの内部表現（最重要設計判断）

**グリッドベース**を採用する。ポリゴン演算（幾何ベース）は面分割のエッジケースが難しく、テストもしにくい。

```
セル状態（2D 配列 Uint8Array）:
  UNCLAIMED   — 未確定領域（QIX が動ける）
  CLAIMED_FAST / CLAIMED_SLOW — 確定エリア（色分け用に 2 値）
  BORDER      — 境界線（マーカー・スパークスが歩ける）
  LINE        — 引き途中のライン（一時状態）
```

- マーカー座標は整数グリッド。1 tick に 1 セル（低速は 2 tick に 1 セル）移動
- 「格子点」と「セル」の二重管理を避けるため、**境界線もセルとして表現する**（線の太さ = 1 セル）。フィールド初期状態は外周 1 セル分が `BORDER`、内側が `UNCLAIMED`

### 4.2 エリア確定アルゴリズム（本作の心臓部）

ラインが境界に到達して閉じたとき:

1. 引いたラインの全セルを `BORDER` に変換
2. QIX の現在位置（頭部セル）から `UNCLAIMED` セルを **flood fill**（4 近傍）
3. fill で到達 **できなかった** `UNCLAIMED` セルをすべて `CLAIMED_FAST` または `CLAIMED_SLOW` に変換
4. 不要になった境界線の掃除: `BORDER` セルのうち、4 近傍 + 斜めに `UNCLAIMED` が 1 つもないもの（= 完全に確定エリアに埋まった線）を `CLAIMED_*` に変換（スパークスの移動範囲を正しく保つため）
5. 占有率を再計算（`CLAIMED_*` セル数 ÷ 初期 `UNCLAIMED` セル数）

実装例（擬似コード）:

```ts
function claimArea(field: Field, line: Point[], qixPos: Point, speed: LineSpeed): ClaimResult {
  for (const p of line) field.set(p, BORDER);
  const reachable = floodFill(field, qixPos, UNCLAIMED); // Set<index>
  let claimed = 0;
  for (const idx of field.indicesOf(UNCLAIMED)) {
    if (!reachable.has(idx)) {
      field.setByIndex(idx, speed === "slow" ? CLAIMED_SLOW : CLAIMED_FAST);
      claimed++;
    }
  }
  pruneDeadBorders(field);
  return { claimedCells: claimed, occupancy: field.occupancy() };
}
```

このアルゴリズムは「QIX がいない側が塗られる」という原作仕様を自然に満たす。**純関数として実装し、ユニットテストの主対象とする。**

**2 匹 QIX への拡張（分断判定）**: ライン閉鎖時に各 QIX の頭部から個別に flood fill する。

- QIX1 の到達集合に QIX2 が含まれない → **分断成功**（即ステージクリア、倍率カウント +1）
- 含まれる場合 → 通常確定。「どの QIX からも到達できない `UNCLAIMED` セル」を塗る（上記 2〜3 の `reachable` を全 QIX の和集合にするだけ）

### 4.3 敵の移動

- **QIX**: 頭部の点が `UNCLAIMED` 内を移動（速度ベクトル + ランダムな角度揺らぎ + `UNCLAIMED` 以外に当たったら反射）。過去 8〜12 tick 分の位置を保持し、各位置に短い線分を描いて「線状生物」を表現。当たり判定は頭部と保持中の各線分に対して行う
- **スパークス**: `BORDER` セル上のグラフ移動。各 tick、現在向きを維持できればを優先し、分岐では「直前に来たセルへ戻らない」制約でマーカー方向に近い分岐を選ぶ（厳密な最短経路は不要。原作も単純追跡）
- **ヒューズ**: プレイヤーが引き途中の `line: Point[]` 配列のインデックスを進むだけ。プレイヤー静止中のみ前進

### 4.4 ゲームループと状態機械

- **固定タイムステップ**（60 updates/sec）+ `requestAnimationFrame` 描画。update と render を分離
- 状態機械: `Title → Playing → (StageClear → Playing) | (Miss → Playing | GameOver) → Title`
- コアロジック（`core/`）は **DOM・Canvas に一切依存しない**。`update(input, dt)` を呼ぶと新しい状態が得られる構造にする（テストの前提条件）

## 5. 入力とレスポンシブ対応

### 5.1 デスクトップ（キーボード）

- 移動: 矢印キー / WASD
- 高速ライン: `X` または `Space` を押しながら移動
- 低速ライン: `Z` または `Shift` を押しながら移動
- ポーズ: `P` / `Esc`

### 5.2 スマホ（タッチ）

- 画面下部に **仮想十字キー（左）+ 2 ボタン（右: FAST / SLOW）** を配置
- `touch-action: none` でスクロール抑止、マルチタッチ対応（移動 + ボタン同時押しが必須要件）
- 縦持ちレイアウト前提。フィールドを上部に、コントロールを下部に

### 5.3 画面スケーリング

- Canvas の内部解像度は論理サイズ固定（例: 640×480、グリッド 160×120 の 4 倍）
- CSS で `aspect-ratio` を維持しつつ画面にフィット（レターボックス）。`devicePixelRatio` 対応で高精細化
- リサイズ・回転イベントで再計算

## 7. テスト戦略

### 7.1 ユニットテスト（Vitest）— 最重要

`core/` は純関数・純クラスなので DOM なしで網羅的にテストする。**特に `claimArea` はゲームの正しさそのもの**なので厚くテストすること。

必須テストケース:

- **field / claimArea**
  - 単純な矩形ラインで面が取れ、QIX のいない側だけが塗られる
  - QIX の位置を左右反転すると、塗られる側も反転する
  - L 字・コの字・渦巻き状ラインでの分割の正しさ
  - フィールド外周に沿った極小の面取り（幅 1 セルの領域）
  - 確定エリアの境界を起点・終点にしたライン
  - 占有率計算: 塗った面積とセル数が一致する / 100% を超えない
  - `pruneDeadBorders`: 確定エリアに埋まった線がスパークスの経路から消える
- **field / claimArea（2 匹 QIX）**
  - 2 匹が同じ側に残る通常確定: どちらからも到達できない領域だけが塗られる
  - 分断成功の検出: ラインで 2 匹が別領域に分かれたら分断クリアと判定される
  - 分断とみなされない境界ケース（ラインが片方の領域を 0 セルにする等）
- **marker**
  - 境界線上のみ自由移動でき、ボタンなしで領域内へ入れない
  - 引き途中ラインとの自己交差が拒否される
  - 低速時に移動レートが半分になる
  - **引き戻し**: 逆順に戻るとラインセルが末尾から消える / 開始点まで戻ると境界線移動に復帰する / ヒューズ出現後は引き戻しできない
- **collision**
  - QIX 頭部/胴体線分 × 引き途中ライン・マーカーの接触判定
  - スパークス × マーカー、ヒューズ × マーカー
  - 確定境界線上のマーカーに QIX が触れてもミスにならない
- **fuse**
  - 静止時間閾値で出現し、静止中のみ前進、エリア確定で消滅する
- **scoring / stage**
  - 高速/低速の得点差、超過ボーナス、ミス時の倍率リセット
  - 分断成功で倍率が `成功回数 + 1` 倍になる / 上限 9 倍で頭打ち / ミスで 1 倍に戻る
  - ステージ遷移で難易度パラメータが正しく変わる（ステージ 3 から QIX が 2 匹）
- **game（状態機械）**
  - Title → Playing → StageClear / GameOver の遷移、ライフ 0 でゲームオーバー

推奨: claimArea 系は **文字列アートでフィクスチャを書ける**ようにする（レビューもしやすい）。

実装例:

```ts
const before = parseField(`
  ##########
  #........#
  #....Q...#
  ##########
`);
const line = pathFrom("(4,0) D D D");     // 上辺から下辺へ縦断
const after = claimArea(before, line, qixAt(before), "slow");
expect(render(after)).toBe(`
  ##########
  #ss #....#
  #ss #Q...#
  ##########
`); // s = CLAIMED_SLOW（QIX のいない左側が塗られる）
```

### 7.2 E2E スモーク（Playwright）

最小限に留める（ゲームプレイの自動 E2E は費用対効果が低い）:

1. ページが読み込まれ Canvas が描画される（スクリーンショット取得）
2. キー入力でタイトル → プレイ開始できる
3. スクリプト化した移動シーケンスで面を 1 つ取り、占有率表示が 0% から増える
4. モバイルビューポートで仮想パッドが表示される

ゲーム内部状態を検証できるよう、`window.__game__`（デバッグフック、production ビルドでは任意）を公開しておくと E2E が安定する。

## 8. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| エリア確定のエッジケース（渦巻き・幅 1 セル領域・境界起点終点） | 誤った側が塗られる/取り残しセル | グリッド + flood fill 方式で単純化。文字列フィクスチャで網羅テスト。**M1 で最優先に固める** |
| スパークスの分岐選択が不自然 | 難易度崩壊（追ってこない/強すぎ） | 厳密性を求めず「戻らない + マーカー寄りの分岐」で十分。手動プレイでチューニング |
| タッチ操作の遊びにくさ | スマホ要件未達 | M5 で実機確認を必須化。仮想パッドのサイズ・位置を config で調整可能に |
| 実装モデルがコアと描画を密結合にする | テスト不能・レビュー困難 | 本書の「core/ は DOM 禁止」を厳守させる。レビューで最初に確認 |
| 引き戻しの状態管理バグ（ライン末尾削除とヒューズ・衝突判定の整合） | ミス判定の不整合 | ラインを単一の `Point[]` で一元管理し、引き戻しは末尾 pop のみで実現。7.1 のテストで担保 |
| 分断判定の境界ケース（分断と同時に面積 0 の領域ができる等） | 誤クリア/クリア漏れ | flood fill ベースの判定に一本化し、文字列フィクスチャでテスト |
| モバイルで SE が鳴らない（自動再生制限） | 体験劣化 | 初回入力での `AudioContext.resume()` を必須実装に（§3.8） |

## 12. v1.1 改善バッチ（2026-07-06 ユーザーフィードバック）

v1 リリース後の実プレイフィードバック 6 件への対応。確認済みの決定: **デバッグパネルはローカル + URL パラメータでのみ有効** / **外周敵の追跡は確率的** / **HUD はフィールドの真上**。

### 12.1 M7: UI レイアウト整列（要望1・3・4）

**変更対象は `src/main.ts`・`src/input/touch.ts`・`index.html`（CSS 含む）のみ。core・render は触らない。**

1. **HUD をフィールド直上に**（要望1）
   - 現状: HUD が viewport 左上に `position: fixed` で孤立し、横長画面でフィールドから離れる（枠と重なることもある）
   - 変更: canvas を包むコンテナをヘッダ行（HUD）+ フィールドの縦積みにし、HUD は**フィールドと同じ幅・真上**に配置。MUTE ボタンは HUD 行の右端に統合
   - リサイズ・レターボックス処理（fitCanvasToViewport）と整合させること（HUD 行の高さを差し引いてフィット計算）
2. **オーバーレイをフィールド中央に**（要望3）
   - 現状: タイトル/クリア/ゲームオーバーのメッセージがページ中央基準
   - 変更: **canvas 要素の中央**に整列（canvas コンテナ基準の absolute 配置）。リサイズ・回転でも追従
3. **タッチパッドの GB 風左右分離**（要望4）
   - 現状: 十字キーと FAST/SLOW が中央寄りで近接
   - 変更: 十字キーを**画面左端**、FAST/SLOW を**画面右端**に寄せ、中央に空間を確保。FAST/SLOW はゲームボーイの A/B のように**斜め配置**（FAST が右上、SLOW が左下）。ボタンサイズ・間隔は誤タッチしない程度に確保（44px 目安以上）

受け入れ基準: §6 M7 行。デスクトップ横長・モバイル縦持ちの両方でスクリーンショット確認。既存 E2E（仮想パッド表示テスト含む）が通ること（セレクタ変更時は E2E 側も追従してよいが、検証内容の緩和は不可）。

### 12.2 M8: 外周敵の確率的追跡（要望2）

**現状の問題**: `Ember.update` が「現在の向きを維持できる限り直進」を最優先するため、外周リング上では永久に直進でき、確定エリアが生んだ内側の境界線（分岐）へ入らない。結果、外周だけを周回して脅威にならない。

**変更（`src/core/patrol.ts`）**:

- 分岐点の定義: 非後退の移動候補が **2 つ以上**あるセル
- 分岐点では、確率 `EMBER_BRANCH_CHASE_PROBABILITY`（**デフォルト 0.7**、config.ts）で**マーカー方向の候補**（dot 積最大）を選び、外れた場合は従来どおり直進優先（直進不可なら dot 積選択）
- 分岐点でない場合は従来挙動（直進優先）を維持
- 乱数は **rng 注入**（コンストラクタに追加。既存シグネチャ互換を保つこと）。`Game` が自分の rng を渡す

**テスト（決定的 rng）**: 内部境界線の分岐を含む文字列フィクスチャで、(a) rng が追跡側のとき分岐でマーカー方向の内側境界線に入る、(b) rng が直進側のとき従来どおり直進する、(c) 非分岐セルでは rng に関係なく直進、を検証。既存テストの期待値は緩めない（挙動変更で正当に変わる場合は理由を報告）。

### 12.3 M9: キャラクター視認性向上（要望6）

**変更対象は `src/render/renderer.ts` と `src/config.ts` のみ。当たり判定・グリッドロジックは一切変更しない**（見た目のサイズと判定は独立）。

| キャラ | 現状 | 変更案 |
|---|---|---|
| 自機（マーカー） | 白 1 セル矩形 | **約 3×3 セル相当のダイヤ型（◆）** + 白グロー。待機中は緩やかにパルス（拡縮） |
| Wisp | 頭部 1 セル + 残像 | 頭部を **約 2×2 セル相当**に拡大（コア + ハロー）。残像は現状維持 |
| Ember | オレンジ 1 セル | **約 2×2 セル相当の火の粉風**（明るいコア + 外周ハロー、ちらつき） |
| Igniter | 赤 1 セル | **約 2×2 セル相当** + 明滅（危険が伝わる速めの点滅） |

- アニメーション位相は renderer 内部のフレームカウンタで（core に状態を持たせない）
- 描画数は少数（自機 1 + 敵数匹）なので shadowBlur 使用可。ただし 60fps 維持（§7.3）
- ミス時グレースの点滅（既存）と干渉しないこと

受け入れ基準: スクリーンショットで各キャラが一目で判別できる。既存テスト・E2E 通過。

### 12.4 M10: デバッグパネル（要望5）

**開発チューニング専用**。プレイヤー向け機能ではない。

- **有効条件**: `import.meta.env.DEV`（Vite dev サーバ時のみ true）**かつ** URL に `?debug`。プロダクションビルドでは dead code として除去されること（dynamic import 推奨: `if (import.meta.env.DEV && params.has('debug')) import('./debug/panel')`）
- **調整項目**（初期セット）:
  1. Wisp の数（0〜4、即時スポーン/除去）
  2. Wisp 速度倍率（0.25〜3.0）
  3. Ember の数（0〜6、即時スポーン/除去）
  4. Ember 移動間隔 tick（1〜10 = 速い〜遅い）
  5. Ember 出現間隔（秒）
  6. 分岐追跡確率（0〜1、M8 の定数）
  7. 要求占有率（10〜90%）
  8. RESET ボタン（ステージ既定値に戻す）
- **設定値のエクスポート（チューニング結果の実装反映用）**:
  - パネルに **EXPORT ボタン**を置き、現在の実効パラメータ一式を **JSON で表示 + クリップボードにコピー**できるようにする
  - JSON のキーは **config.ts の定数名と一致**させる（例: `{"WISP_SPEED": 0.45, "EMBER_MOVE_TICKS": 2, "EMBER_BRANCH_CHASE_PROBABILITY": 0.8, ...}`）。定数に直接対応しないステージ依存項目（Wisp 数等）は対応する config 定数名（`STAGE3_WISP_COUNT` 等）か注記付きキーで出す
  - 運用フロー: パネルで調整 → EXPORT → JSON を会話に貼る → 「軽微な修正」としてサブエージェントが config.ts に反映（キー名が一致しているので機械的に適用できる）
- **即時反映の仕組み**: `src/core/` に**実行時オーバーライド注入口**を追加する（例: `Game` に `applyDebugOverrides(overrides)` またはオーバーライドオブジェクトの setter）。core は DOM 非依存のまま（パネル UI は `src/debug/` に置き、main.ts が結線）。オーバーライドは**ステージをまたいで維持**し、RESET で解除
- **テスト**: core 側オーバーライド API のユニットテスト（数の増減・速度反映・ステージ遷移後の維持・解除）。パネル UI は手動確認でよい
- HUD/画面に「DEBUG」表示を出し、デバッグ中であることが分かるようにする（ハイスコアの汚染を避けるため、**デバッグパネル使用中はハイスコアを保存しない**）
