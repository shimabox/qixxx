## P2: core 割り当て削減の結果（2026-07-07）

計測方法: core は純粋ロジックなので Node 単体で決定的ベンチマークを実行できる。固定シード rng + スクリプト入力でステージ 10 相当（Wisp10×速度5・Ember10×moveTicks1）の Game を 120,000 tick 回し、`node --trace-gc` の Scavenge 回収量合計 = 実割り当て量を before/after 比較した（ドライバは esbuild で core をバンドルしたもの。シミュレーションは決定的なので、最終スコア/占有率/ゲームオーバー回数の一致 = 挙動不変の証明になる）。

| 指標 | 改修前 | 改修後 | 変化 |
|---|---|---|---|
| 短命割り当て（120k tick 合計） | 6,775.7 MB（56.5KB/tick） | 2,370.1 MB（19.8KB/tick） | **-65%** |
| Scavenge GC 回数 | 461 | 152 | **-67%** |
| GC 総時間 | 110 ms | 42.8 ms | -61% |
| スループット | 54,246 ticks/s | 92,043 ticks/s | +70% |
| 挙動（score / occupancy / rebuilds） | 43 / 0.004666380605020382 / 233 | 同値 | **完全一致** |

変更内容は plan.md §13.3 P2 の 3 項目 + 着手前計測で支配的と判明した patrol.ts（Ember.update の map/filter チェーン→ビットマスク単一ループ化）。テストは 147→152 件（+5）全通過。

計測時の学び: Node `--heap-prof`（サンプリングヒーププロファイル）は生存オブジェクト偏重で短命ゴミの帰属には使えなかった。`--trace-gc` の Scavenge 回収量合計が churn のグラウンドトゥルースとして機能した。

残課題（P3 以降): 描画層の毎フレーム割り当て（renderFrame の getTrail クローン・rgba 文字列・HUD textContent）。また `Marker.getLine()` がライン全体をクローンするため、Igniter 動作中は毎 tick + 毎フレーム（getIgniterPosition 経由）にライン長ぶんの割り当てが発生する — P3 で対象に含める候補。

# M13 P1: ソークテスト ベースライン計測（2026-07-07）

docs/plan.md §13.3 P1 の実測記録。**結論: 実ブラウザ 11.7 分の最難条件ソークでメモリリークの兆候なし。** §13.1 の静的調査の結論（真のリークは無い）が動的計測でも裏付けられた。一方で V8 確保済みヒープの膨張（5.6 → 最大 34.4MB）が観測され、§13.2-A の GC チャーン（割り当てレート）削減が P2/P3 で取り組む価値のある実在の課題であることも確認できた。

## 計測方法

- 環境: Chrome（chrome-devtools MCP 経由）、Vite dev サーバ `http://localhost:5173/?debug`
- 条件: デバッグオーバーライドでステージ 10 相当の最難パラメータを適用
  `{ wispCount: 10, wispSpeedMultiplier: 5.0, emberCount: 10, emberMoveTicks: 1, emberBranchChaseProbability: 1.0, emberSpawnIntervalSec: 5 }`
- 自動化ハーネス（`evaluate_script` で注入）:
  - 入力ジグラー: 350ms ごとにランダム方向キー + 70% の確率で描画ボタン（ライン引き・claimArea・衝突判定・ミスの経路を実際に通す）
  - 自動リスタート: 非 playing 状態を検知したら confirm キーを送出（Title/GameOver/StageClear を自動通過）
  - サンプラー: 5 秒ごとに `performance.memory` を記録
- 前後でヒープスナップショット取得（取得時にフル GC が走るため、GC 後の実保持量を比較できる）

## 結果

| 指標 | 値 | 判定 |
|---|---|---|
| ソーク時間 / サンプル数 | 700 秒 / 140 点 | — |
| 画面遷移（ゲームオーバー→リスタート等） | 25 回 | Game/Field の作り直し経路も反復実行 |
| used ヒープの推移 | 2.7〜15.0MB の鋸歯状。極小値（GC 後の床）は 2.7〜5.7MB を往復し**単調増加なし** | ✅ リーク兆候なし |
| GC 後の実保持量（スナップショット強制 GC 後） | 開始 2.99MB → 終了 3.68MB（**+0.69MB**、計測ハーネス自身の 140 サンプル配列等を含む） | ✅ 横ばい |

## 解釈と M13 への含意

1. **リークは無い**: GC 後の床が 10 分間フラットで、リスタートを 25 回繰り返しても到達不能オブジェクトの残留がない。§13.1 の有界性分析どおり。
2. **割り当てレートは実際に高い**: V8 が確保ヒープを 6 倍超に広げたのは、鋸歯の振幅（1 GC サイクルあたり最大 10MB 超の短命割り当て）に追従したため。これはフレームスパイクの潜在要因であり、P2（core のクローン削減）/ P3（render の文字列・配列割り当て削減）の実施根拠になる。
3. **未取得データ**: Allocation sampling（割り当て元 Top のプロファイル）は MCP 経由で取得できないため未実施。P2 着手時に DevTools の Allocation instrumentation を手動で当てて、静的調査で特定済みの候補（`getTrail()` クローン・`cellsBetween` 配列・rgba 文字列）と突き合わせるのが望ましい。
4. 改善後（P2/P3 完了時）に同条件で再計測し、total ヒープの最大値と鋸歯の振幅が下がることを確認する。

## 再現手順（次回計測用）

1. `npm run dev` → `http://localhost:5173/?debug` を Chrome で開く
2. DevTools コンソールで `window.__game__.session.applyDebugOverrides({...上記パラメータ})`
3. 上記ハーネス（入力ジグラー / 自動リスタート / 5 秒サンプラー）を注入し 10 分放置
4. 前後でヒープスナップショットを取り、GC 後 used とインスタンス数（Game/Field/Wisp/Ember/OscillatorNode）を比較
